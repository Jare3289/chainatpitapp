# Ultimate System Review & Technical Deep Dive (CNP APP)

เอกสารฉบับนี้เป็นการสรุปภาพรวมและเจาะลึกการทำงานของระบบ CNP APP ทั้งหมด ตั้งแต่ตรรกะเบื้องหลังไปจนถึงการแสดงผลหน้าบ้าน เพื่อใช้เป็นคู่มืออ้างอิงสูงสุดสำหรับทีมพัฒนา

---

## 📋 1. มิติของตรรกะธุรกิจ (Business Logic Review)

### 👤 1.1 ระบบบริหารจัดการสิทธิ์ (Role-Based Access Control)
ระบบแบ่งผู้ใช้ออกเป็น 3 ระดับหลัก โดยมีขอบเขตการเข้าถึงข้อมูลที่ต่างกัน:
- **Admin**: เข้าถึงทุกเมนู จัดการฐานข้อมูลนักเรียน/ครูได้ทั้งหมด และนำเข้าข้อมูลแบบ Bulk Import
- **Teacher**: จัดการข้อมูลส่วนตัว และดูข้อมูลนักเรียนใน "ห้องที่ปรึกษา (Advisory Room)" ของตนเองได้เท่านั้น
- **Student**: จัดการข้อมูลส่วนตัว และดูประวัติการมาเรียน/คะแนน/ตารางสอนของตนเอง

### 🏫 1.2 ระบบคณะสี (House System)
เป็นหัวใจสำคัญของการสร้าง Visual Identity ในระบบ:
- **Data Layer**: ใช้ ENUM ในฐานข้อมูล (`ขุนสรรค์`, `ขุนศรี`, `เจ้ายี่`, `ธรรมจักร`) เพื่อป้องกันข้อมูลผิดเพี้ยน
- **Logic Layer**: JavaScript Mapping แปลงชื่อคณะสีเป็นรหัสสี (Hex Code) เพื่อใช้ใน UI
- **Visual Layer**: เมื่อเปลี่ยนคณะสี ธีมของหน้าเว็บ (CSS Variables) จะเปลี่ยนตามทันที

### 1.3 ตรรกะการตรวจสอบความสมบูรณ์ (Profile Completion Logic)
ระบบมีการทำ **Data Audit** อัตโนมัติในหน้าโปรไฟล์นักเรียน:
1. กำหนด **Essential Fields** (เช่น เลขบัตรประชาชน, เบอร์โทร, ข้อมูลครอบครัว)
2. เมื่อโหลดข้อมูล JavaScript จะวนลูปเช็คว่าฟิลด์เหล่านี้มีค่าหรือไม่ (รองรับการเช็คค่าว่างและเครื่องหมาย "-" เพื่อความแม่นยำ)
3. คำนวณเปอร์เซ็นต์แบบถ่วงน้ำหนัก (Weighted Completion)
4. แสดงผลผ่าน **SVG Circular Progress** พร้อมรายการ "ข้อมูลที่ยังขาดหาย" เพื่อให้ผู้ใช้ทราบจุดที่ต้องแก้ไข

## 🛠️ 2. มิติของงานวิศวกรรมซอฟต์แวร์ (Technical Engineering)

### 🏗️ 2.1 โครงสร้างการประสานงาน (Module Coordination)
การทำงานร่วมกันระหว่างไฟล์ต่างๆ ถูกออกแบบมาให้เป็นระเบียบ:
- **`config.php`** ➔ **`inc/db.php`**: เริ่มต้นการเชื่อมต่อ
- **`inc/security.php`**: ถูกเรียกใช้ในทุก API เพื่อทำหน้าที่เป็น "ยาม" ตรวจสอบสิทธิ์
- **`api/*.php`**: ทำหน้าที่เป็น "คนส่งสาร" รับคำขอจาก JS และคุยกับ DB
- **`views/*.html`**: ทำหน้าที่เป็น "หน้าด่าน" รับ Input และแสดง Output

### 🚀 2.2 ระบบการนำเข้าข้อมูล (Bulk Data Processing)
ในหน้า Admin มีระบบนำเข้าข้อมูลที่ซับซ้อน:
- รองรับไฟล์ **Excel (.xlsx)** และ **CSV**
- มีขั้นตอน **Pre-validation**: ตรวจสอบข้อมูลซ้ำ (Duplicate Check) ก่อนบันทึกจริง
- ใช้ **Transaction (PDO)** ในการบันทึก เพื่อให้มั่นใจว่าหากเกิดข้อผิดพลาด ข้อมูลจะไม่พัง (Rollback)

## 🎨 3. มิติของ UI/UX และ Aesthetics

### 💎 3.1 Design Language
- **Typography**: ใช้ `Google Sans` และ `Inter` เพื่อความเป็นทางการแต่ยังดูสมัยใหม่
- **Glassmorphism**: ใช้ในส่วนของ Card และ Sidebar เพื่อเพิ่มความลึก (Depth) ให้กับ UI
- **Micro-animations**: ใช้ CSS Transitions ในการเปลี่ยนสถานะปุ่มและ Modal เพื่อให้ User Experience ลื่นไหล

### 3.2 Responsive Navigation & Interaction
- **Desktop**: Collapsible Sidebar ที่จดจำสถานะการเปิด/ปิด
- **Mobile**: Bottom Navigation Bar สไตล์แอปมือถือ เพื่อให้เข้าถึงเมนูหลักได้ง่ายด้วยนิ้วโป้ง
- **Click-to-Call**: เบอร์โทรศัพท์ในระบบจะถูกแปลงเป็น Link (`tel:`) อัตโนมัติใน View Mode เพื่อให้แอดมินโทรออกได้ทันที

## 💾 4. มิติของฐานข้อมูล (Database Deep Dive)

### 🔑 4.1 ความสัมพันธ์ของข้อมูล (Data Relations)
- **Users <-> Entities**: ตาราง `users` เชื่อมโยงกับ `students` หรือ `teachers` ผ่าน `user_id`
- **Advisory Mapping**: ครูเชื่อมโยงกับห้องเรียนผ่าน `advisory_room_id`
- **Attendance**: เชื่อมโยงนักเรียนรายบุคคลเข้ากับวันที่เรียนและสถานะการมา

## 🛡️ 5. สรุปมาตรการรักษาความปลอดภัย (Security Summary)

| ประเภทภัยคุกคาม | วิธีการป้องกันในระบบ |
| :--- | :--- |
| **SQL Injection** | ใช้ PDO Prepared Statements 100% |
| **XSS Attack** | มีการ Escape HTML Output และ Sanitization |
| **CSRF Attack** | ใช้ CSRF Token ในทุกคำขอที่มีการเปลี่ยนแปลงข้อมูล |
| **Brute Force** | ระบบ Hash รหัสผ่านแบบ Bcrypt และการเช็ค Session |

---

*ระบบ CNP APP ถูกสร้างขึ้นโดยเน้นที่ความยืดหยุ่น (Scalability) เพื่อรองรับการเพิ่มฟีเจอร์ใหม่ๆ ในอนาคต เช่น ระบบตัดคะแนนพฤติกรรม หรือระบบขอใบรับรองออนไลน์*
